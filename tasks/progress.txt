## Codebase Patterns
- New SQLite tables need both a `src/**/*.sql.ts` definition and a matching export from `packages/opencode/src/storage/schema.ts` so `Database.Client()` and Drizzle migrations stay aligned.
- Delivery-state mutations belong in `packages/opencode/src/delivery/store.ts`; SharedBoard and UI code should consume SQLite-backed read models instead of writing delivery tables directly.
- `DeliveryStore.launch()` is the authoritative way to seed a recoverable swarm delivery run: it persists `swarm_run.phases`, ensures default delivery roles exist, and inserts run-linked staged `work_item` rows before conductor execution continues.
- Delivery phase gates live in `swarm_run.gate` and `work_item.gate`; use `DeliveryStore.evaluateRun()` or the store write paths to refresh blocker reasons before advancing `swarm_run.phase`.
- Use `DeliveryStore.reviewAssignments()` before changing role plans or `owner_role_id`; `DeliveryStore.assign()` only applies same-role scheduling updates and raises `DeliveryAssignmentError` for role add/remove, responsibility changes, or cross-role owner reassignments.
- Major role-plan deltas should go through `DeliveryStore.requestAssignment()` and `DeliveryStore.resolveAssignment()`; store the before/after proposal in `open_question.context` and keep the linked `Decision` in `proposed` until the question is resolved.
- Use `DeliveryStore.recordDecision()` for structured consensus activity; keep proposal-time `participants`, `candidate_outcomes`, and `input_context` on the authoritative `decision` row so later review, objection, and final decision actions append to one log.
- Non-confirmation decisions resolve inside `DeliveryStore.recordDecision()`: all listed participants must have recorded outcomes before consensus or conductor arbitration can decide the row, and the terminal `decision` action `context` stores JSON `{ reason, updates }` for downstream consumers.
- Route open-question lifecycle changes through `DeliveryStore.updateQuestion()`; it is responsible for clearing `blocking` on resolved/deferred/cancelled questions, syncing the linked `Decision`, and re-blocking deferred questions when the affected work-item gate becomes current.
- Work-item recovery state lives in `work_item.checkpoint` and `work_item.failure`; keep successful progress in `checkpoint`, let `updateItem()` snapshot verify/commit failures into `failure`, and only flip `destructive_cleanup_allowed` through `DeliveryStore.allowCleanup()` with a decided `destructive_cleanup` decision.
- `DeliveryStore.resume()` is the authoritative recovery entrypoint: it reloads SQLite run/items first, rewinds interrupted `in_progress` or `verifying` work back to a resumable state, preserves verify/commit failure evidence, and opens a blocking `recovery_review` question when checkpoint or failure metadata conflicts with current delivery state.
- Keep `small_mr_required` enabled for P1 delivery work items and let `DeliveryStore.sync()` block the run whenever more than one `implement`/`verify`/`commit` item is active; the single-thread guard is the authoritative small-MR enforcement point.
- Complete the commit gate through `DeliveryStore.recordCommit()` rather than `updateItem()`; it is the only path that writes `work_item.commit`, requires passed verification proof plus staged scope, and appends the matching `swarm_run.audit` commit event.
- `swarm_run.audit` is the structured run history for phase, assignment, decision, question, verification, and commit events; use `DeliveryStore.lookup()` or `DeliveryStore.resume()` to rebuild completed versus pending work and keep already verified or committed items out of the resume queue.
- Finalize delivery runs through `DeliveryStore.recordRetrospective()`; it records linked retrospective output on `swarm_run`, appends a `retrospective` audit event, writes durable memories through the project memory store, and blocks direct terminal `updateRun(...status)` writes until retrospective exists.
- Delivery API consumers should use `DeliveryStore.read()` via `/swarm/:id/delivery`; it refreshes run gates from SQLite and returns one authoritative payload for the run, work items, decisions, questions, blockers, and small-MR state.
- Swarm delivery UI should use the dedicated `/:dir/swarm/:id/run` route backed by `/swarm/:id/delivery`; keep `/:dir/swarm/:id` focused on admin controls so run monitoring stays separate from session and admin views.
- Swarm run action panels should call the delivery write routes and then refetch the same `/swarm/:id/delivery` resource; keep decisions and questions authoritative instead of staging local optimistic copies.
- Swarm run ship-status and audit sections should read the authoritative `state`, `commit`, and `audit` fields from `/swarm/:id/delivery`; avoid reconstructing delivery history from EventSource snapshots in client state.
