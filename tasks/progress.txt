## Codebase Patterns
- New SQLite tables need both a `src/**/*.sql.ts` definition and a matching export from `packages/opencode/src/storage/schema.ts` so `Database.Client()` and Drizzle migrations stay aligned.
- Delivery-state mutations belong in `packages/opencode/src/delivery/store.ts`; SharedBoard and UI code should consume SQLite-backed read models instead of writing delivery tables directly.
- `DeliveryStore.launch()` is the authoritative way to seed a recoverable swarm delivery run: it persists `swarm_run.phases`, ensures default delivery roles exist, and inserts run-linked staged `work_item` rows before conductor execution continues.
- Delivery phase gates live in `swarm_run.gate` and `work_item.gate`; use `DeliveryStore.evaluateRun()` or the store write paths to refresh blocker reasons before advancing `swarm_run.phase`.
- Use `DeliveryStore.reviewAssignments()` before changing role plans or `owner_role_id`; `DeliveryStore.assign()` only applies same-role scheduling updates and raises `DeliveryAssignmentError` for role add/remove, responsibility changes, or cross-role owner reassignments.
- Major role-plan deltas should go through `DeliveryStore.requestAssignment()` and `DeliveryStore.resolveAssignment()`; store the before/after proposal in `open_question.context` and keep the linked `Decision` in `proposed` until the question is resolved.
- Use `DeliveryStore.recordDecision()` for structured consensus activity; keep proposal-time `participants`, `candidate_outcomes`, and `input_context` on the authoritative `decision` row so later review, objection, and final decision actions append to one log.
- Non-confirmation decisions resolve inside `DeliveryStore.recordDecision()`: all listed participants must have recorded outcomes before consensus or conductor arbitration can decide the row, and the terminal `decision` action `context` stores JSON `{ reason, updates }` for downstream consumers.
- Route open-question lifecycle changes through `DeliveryStore.updateQuestion()`; it is responsible for clearing `blocking` on resolved/deferred/cancelled questions, syncing the linked `Decision`, and re-blocking deferred questions when the affected work-item gate becomes current.
