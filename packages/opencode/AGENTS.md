# opencode database guide

## Database

- **Schema**: Drizzle schema lives in `src/**/*.sql.ts`.
- **Runtime exports**: After adding a new table in `src/**/*.sql.ts`, also export it from `src/storage/schema.ts` so `Database.Client()` uses the same schema set as Drizzle migrations.
- **Delivery schema**: Keep delivery enums and validators in `src/delivery/schema.ts`, reference them from `src/delivery/delivery.sql.ts`, and extend `test/storage/delivery-schema.test.ts` when adding delivery-state tables.
- **Delivery store**: Route `SwarmRun`, `RoleSpec`, `WorkItem`, `Decision`, and `OpenQuestion` writes through `src/delivery/store.ts`; keep SharedBoard and UI state projection-only relative to SQLite.
- **Delivery assignment review**: Use `DeliveryStore.reviewAssignments()` or `DeliveryStore.assign()` for role-plan and work-item owner changes; major deltas (role add/remove, responsibility change, cross-role owner reassignment) raise `DeliveryAssignmentError` instead of mutating SQLite.
- **Delivery assignment confirmation**: Major role-plan deltas should flow through `DeliveryStore.requestAssignment()` and `DeliveryStore.resolveAssignment()`; keep the proposed before/after payload in `open_question.context` and do not mutate roles or work items until the linked `Decision` is resolved.
- **Delivery decision actions**: Use `DeliveryStore.recordDecision()` to append `proposal`, `review`, `objection`, and `decision` actions to the authoritative `decision.actions` log; `proposal` entries also own `participants`, `candidate_outcomes`, and `input_context` on the same row.
- **Delivery decision resolution**: `DeliveryStore.recordDecision()` auto-resolves non-confirmation decisions when every listed participant has a recorded outcome and they align; conductor `decision` actions can arbitrate informed non-confirmation conflicts, and the terminal action `context` should store JSON `{ reason, updates }` metadata for the resolved targets.
- **Delivery question lifecycle**: Route question lifecycle changes through `DeliveryStore.updateQuestion()`; terminal or downgraded states auto-clear `blocking`, sync the linked `Decision` (`resolved -> decided`, `deferred -> superseded`, `cancelled -> cancelled`), and deferred questions should target the future `WorkItem` ids that must be re-checked when that gate becomes current.
- **Delivery checkpoints**: Store work-item recovery state in `work_item.checkpoint` and `work_item.failure`; update it through `DeliveryStore.updateItem()`, and only unlock destructive cleanup with `DeliveryStore.allowCleanup()` after a decided `destructive_cleanup` `Decision` applies to that work item.
- **Delivery small MR**: `small_mr_required` stays `true` for P1 delivery work, and `DeliveryStore.sync()` should block the run when more than one `implement`/`verify`/`commit` item is active so commit stays single-threaded through one story path.
- **Delivery commit writeback**: Finish the commit gate through `DeliveryStore.recordCommit()` instead of `updateItem()`; it is the shared path that snapshots `work_item.commit`, requires passed verification proof plus staged scope, and appends the matching `swarm_run.audit` commit event.
- **Delivery retrospective**: Finalize delivery runs through `DeliveryStore.recordRetrospective()`; direct terminal `updateRun(...status)` writes are blocked until retrospective output is stored, and high-impact retrospective memories should be persisted as `pending` for human confirmation.
- **Delivery audit lookup**: `swarm_run.audit` is the structured per-run history for phase, assignment, decision, question, verification, commit, and retrospective events; use `DeliveryStore.lookup()` or `DeliveryStore.resume()` to rebuild completed vs pending work from audit-backed state before re-queueing verified or committed items.
- **Delivery API read model**: Expose delivery UI and SDK reads through `DeliveryStore.read()` and `/swarm/:id/delivery`; that route refreshes gates from SQLite and returns the authoritative run, items, decisions, questions, blockers, and small-MR state without reconstructing free-text logs.
- **Delivery API writes**: Keep delivery confirmation and question-action routes thin: confirm role changes through `DeliveryStore.resolveAssignment()`, move question actions through `DeliveryStore.updateQuestion()`, and always return a fresh `DeliveryStore.read()` payload after the mutation.
- **Delivery resume**: Use `DeliveryStore.resume()` to rebuild a run from SQLite state; it resets interrupted `in_progress` or `verifying` work back to a resumable state, keeps verify/commit failure evidence, and raises a blocking `recovery_review` decision plus open question when checkpoint or failure metadata conflicts with the authoritative run state.
- **Delivery launch**: Use `DeliveryStore.launch()` to seed a recoverable delivery run; it owns the ordered `swarm_run.phases` list, default role specs, and run-linked staged `work_item` rows for `Swarm.launch()`.
- **Delivery gates**: Persist per-phase gate JSON on both `swarm_run` and `work_item`; use `DeliveryStore.evaluateRun()` or the write paths in `src/delivery/store.ts` to refresh blocked reasons before advancing `swarm_run.phase`.
- **Naming**: tables and columns use snake*case; join columns are `<entity>_id`; indexes are `<table>*<column>\_idx`.
- **Migrations**: generated by Drizzle Kit using `drizzle.config.ts` (schema: `./src/**/*.sql.ts`, output: `./migration`).
- **Command**: `bun run db generate --name <slug>`.
- **Output**: creates `migration/<timestamp>_<slug>/migration.sql` and `snapshot.json`.
- **Tests**: migration tests should read the per-folder layout (no `_journal.json`).

## Swarm State

- Use `SwarmState.mutate` for authoritative swarm, task, and discussion writes; the per-task and per-discussion JSON files are compatibility projections derived from `board/<swarm>/state.json`.
- When adding canonical swarm snapshot fields, update `SwarmState.Alignment` or the root `Snapshot` shape in `src/session/swarm-state.ts`, and initialize matching defaults in both `SwarmState.Example` and `SwarmState.create()`; current rollout rejects older snapshot schema versions instead of adding compatibility loaders.
- Project-scoped alignment persistence lives in `board/alignment.json`; keep shared role definitions in `alignment.catalog.roles` and user-specific approvals in `alignment.confirmations.users` so future writes do not mix shared catalog state with per-user confirmation state.
- `SwarmState.classify()` is the material role-delta checker: it matches requested roles by `role_id` first, falls back to `name`, and only treats `purpose`, `perspective`, and `default_when` as meaningful changes after whitespace normalization.
- Catalog-backed run roles should leave overlay fields `null`; `SwarmState.classify()` treats those `null` values as inherited catalog values instead of false-positive modifications.
- `delegate_task` is the first guaranteed pre-worker checkpoint for swarm runs; persist or refresh `alignment.contract` there via `SwarmState.draft()` before the worker session is created or registered.
- Persist user-facing contract copy in `alignment.summary` and render it with `SwarmState.renderSummary()` so CLI and web surfaces reuse the same non-prompt summary.
- Use `SwarmState.renderDecision()` when a CLI flow needs gate context plus the stored summary; it prepends the gate and reason without exposing internal worker prompts.
- `SwarmState.decide()` owns deterministic gate mapping: sensitivity => `G3`, material role delta => `G2`, ambiguity/trade-offs => `G1` or `G2` depending on confidence and novelty, then confidence and routine-vs-novel decide the fallback between `G1` and `G0`.
- `SwarmState.admit()` is the discussion threshold gate: only the three primary signals (`multiple_valid_options`, `meaningful_trade_offs`, `direction_change`) count toward the 2-of-3 admission rule; `role_benefit` is advisory only.
- Keep gate and discussion truth-table coverage in `test/session/swarm-state.test.ts`; the policy helpers are pure there, so new thresholds should be asserted directly instead of through conductor integration.
- `SwarmState.preflight()` is the conductor-ready alignment bundle: it drafts the contract, classifies role deltas, evaluates the gate, and decides whether delegation can proceed or must pause with `pending_confirmation`.
- Use `SwarmAdmin.readAlignment()` and `SwarmAdmin.get()` for integration coverage after approval or escalation transitions; they are the server-side consumer contracts that should stay readable after state changes.
- High-risk runs reuse `alignment.run_confirmation` only when the stored contract and gate still match; `Swarm.confirm()` clears `pending_confirmation`, refreshes the summary, and `Swarm.resume()` should reject manual resumes while a run confirmation is still pending.
- Use `SwarmState.writeback()` for approved role changes: it only rewrites `modified` or `added` deltas, merges null overlay fields back to the catalog baseline, and records both catalog audit metadata and per-user confirmation state.
- `SwarmState.preflight()` accepts gate-input overrides for later phases; if the new gate outranks the stored gate, it emits an escalation-specific pending-confirmation reason and any stale `run_confirmation` should be cleared before delegation continues.
- Server-side alignment reads should use `SwarmAdmin.Alignment`: expose it directly at `/swarm/:id/alignment` for CLI consumers and embed the same read model in `SwarmAdmin.Detail` so web surfaces do not reconstruct alignment state client-side.
- Worker timeout behavior comes from `swarm.wait_timeout_seconds` in config and is mirrored into each launched swarm's config; when testing wait behavior, use `Swarm.status()` to trigger the timeout promotion from `waiting` to `blocked`.
- Before launching v3 swarms in a workspace, run `opencode swarm-cleanup --dry-run` and then `opencode swarm-cleanup --confirm purge-legacy-swarms`; `Swarm.launch()` is intentionally blocked until that cleanup marker exists.
